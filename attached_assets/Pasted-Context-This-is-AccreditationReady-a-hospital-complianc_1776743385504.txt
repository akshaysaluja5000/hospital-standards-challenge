===========================================
1. ROLE DEFINITIONS (source of truth)
===========================================

Create a single roles config file (e.g. /shared/roles.ts or /server/config/roles.ts) that is the ONLY source of truth for the role cards, routing, and first-module assignment. The role-select page, router, and admin views must all read from this file. Schema:

{
  id: string,                     // stable slug, e.g. "or_circulating_nurse"
  department: string,             // e.g. "Operating Room"
  title: string,                  // e.g. "OR Circulating Nurse"
  description: string,            // short role description shown on card
  scope: "DEPT" | "DEPT_PLUS_ALL" | "FULL",
  destinationRoute: string,       // e.g. "/training/or-nurse"
  firstModuleId: string,          // id of the first quiz/module shown
  reportingScope: "own_department" | "own_plus_all" | "enterprise"
}

===========================================
2. ROUTING REQUIREMENTS
===========================================

- Tapping Continue must route to the destinationRoute for the selected role's id. No hard-coded route strings outside the config.
- The destination page MUST visibly reflect the chosen role in its title/header (e.g. "Training — OR Circulating Nurse") and MUST render the firstModuleId as the first question/module.
- If a role id is missing, unknown, or the module lookup fails, render an explicit error page with a "Change role" button, NOT a blank screen or a silent fallback.

===========================================
3. SELECTION AND VALIDATION BEHAVIOR
===========================================

- Role cards must have a clearly distinct selected state (border + background + checkmark or equivalent). Do not rely on color alone.
- Only one role can be selected at a time. Rapidly tapping a second card must cleanly replace the first selection with no race condition.
- Continue is disabled until a role is selected. If a user tries to continue without a selection (keyboard, screen reader, or any bypass), show an inline error above the Continue button: "Select a role to continue." The existing "No role selected" footer text is too subtle on its own — keep it but add the inline error.
- Selected role must persist in session state so a page refresh returns the user to the same selected card (not wiped). Browser back from the destination page must return to the role-select page with the previous selection still highlighted.
- Add a visible "Change role" affordance on the destination page so users are never trapped.

===========================================
4. ACCESS SCOPE CHIPS — MAKE THEM MEAN SOMETHING
===========================================

Right now "DEPT. ACCESS", "DEPT + ALL", and "FULL ACCESS" are opaque. Do two things:

a) Add a tooltip / tap-to-expand on each chip with plain-English text:
   - DEPT. ACCESS   → "You'll see training and reporting for your department only."
   - DEPT + ALL     → "You'll see your department plus a view across all departments."
   - FULL ACCESS    → "You'll see training and reporting for the entire organization."

b) Enforce the scope on the destination and on any admin/reporting views:
   - DEPT roles see only their own department's data.
   - DEPT_PLUS_ALL roles see their department plus a read-only all-department view.
   - FULL roles see the enterprise dashboard.
   Scope must be enforced server-side, not just hidden in the UI.

===========================================

===========================================
6. EDGE CASES TO HANDLE EXPLICITLY
===========================================

Write handlers (not just hope) for:
- Continue tapped with no selection → inline error (see §3).
- Role A selected, then Role B tapped, then Continue → routes to Role B's destination.
- Refresh after selecting → selection persists.
- Browser back from destination → returns to role-select with the previous selection visible and re-selectable.
- Role changed from profile after training has started → user is asked "Switch roles? Your current progress will be saved under [Old Role]." before the switch commits.

===========================================
7. ACCEPTANCE CRITERIA (treat as tests I will run)
===========================================

The build is not pilot-ready until ALL of these pass:

1. Every role in the config routes to its declared destinationRoute, and the destination shows the declared firstModuleId as the first module.
2. A first-time user reaches the first question of their role's training in no more than two taps from the role-select page.
3. Pressing Continue with no selection produces a visible inline error, not a silent no-op.
4. Leadership roles (Compliance Officer / CNO, Nurse Educator) visibly land on a broader view than DEPT-only roles — not the same page with hidden differences.
5. Refreshing the role-select page or using browser back does not wipe the user's selection.
6. Every scope chip has a readable explanation on hover/tap.
7. Nothing in the UI references route paths, role ids, or internal labels — only MOSH-native, human language.
